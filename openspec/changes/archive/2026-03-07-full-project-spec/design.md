## Context

keketrade は日本株専用の投資支援 CLI ツール。旧リポジトリ kekestock（18 コマンド）を Rust 2024 edition で再設計し、5 コマンドパイプライン + サブコマンド構成に整理した。

現在の実装は Phase 1〜3 が完了しており、全コマンドが動作する状態。Tachibana Securities API 連携のみスタブ（API アクセス待ち）。

制約:
- JP 市場のみ（US 株は対象外）
- 個人利用ツール（マルチユーザー不要）
- CLI ファースト（GUI なし）
- ログは stderr、データ出力は stdout（パイプライン親和）

## Goals / Non-Goals

**Goals:**
- LLM を活用した日本株の定量・定性評価パイプラインの提供
- テクニカル指標とファンダメンタル情報の統合的な投資判断
- 安全な自動売買（サーキットブレーカー + ドライランデフォルト）
- 投資戦略の外部 YAML 管理と再現性（SHA256 ハッシュ追跡）
- JSON 出力によるスクリプタビリティ（cron/launchd 自動化対応）

**Non-Goals:**
- リアルタイムトレーディング（日次バッチ処理のみ）
- US 株・暗号資産など JP 市場以外の対応
- Web UI / モバイルアプリ
- マルチユーザー / 認証・認可
- バックテスト機能（将来検討）

## Decisions

### 1. SQLite (tokio-rusqlite, bundled) を採用

**選択**: SQLite with bundled feature
**理由**: 個人利用ツールのため外部 DB サーバー不要。`bundled` で環境依存を排除。`tokio-rusqlite` で async コンテキストから同期 SQLite を安全に利用。
**代替案**: PostgreSQL（過剰）、sled（クエリ表現力不足）、ファイル直書き（検索性不足）

### 2. 金額を TEXT (rust_decimal) で保存

**選択**: `rust_decimal::Decimal` を TEXT カラムに保存
**理由**: 浮動小数点の丸め誤差を回避。金融計算の正確性を担保。serde-with-str feature で JSON も文字列として出力。
**代替案**: f64（丸め誤差あり）、INTEGER で最小単位管理（桁管理が煩雑）

### 3. LLM バックエンド抽象化（trait + factory）

**選択**: `LlmBackend` trait + `create_backend()` factory パターン
**理由**: CLI ベース（claude/gemini コマンド）と HTTP API ベースの 4 バックエンドを統一的に扱う。設定ファイルの文字列で切替可能。
**代替案**: enum ディスパッチ（拡張性低い）、設定なし固定（柔軟性不足）

### 4. 冪等書き込み（INSERT OR IGNORE / ON CONFLICT）

**選択**: 全 DB 書き込みを冪等に設計
**理由**: パイプラインの再実行やリトライに安全。同じデータを複数回書いてもエラーにならない。
**代替案**: UPSERT のみ（意図しない上書きリスク）、事前チェック（レースコンディション）

### 5. 投資 Spec を外部 YAML で管理

**選択**: `~/.config/kabu/specs/*.yaml` に戦略パラメータを定義し、SHA256 ハッシュを評価に記録
**理由**: 戦略変更の追跡可能性。eval 結果にどの Spec バージョンで評価したかを紐付け。YAML は人間が読み書きしやすい。
**代替案**: config.toml に埋め込み（肥大化）、DB 管理（バージョン管理しにくい）

### 6. サーキットブレーカーによる安全制御

**選択**: execute 前に自動チェック。個別銘柄 >30% 変動 or ウォッチリスト >50% が >5% 下落でブロック
**理由**: 異常相場での意図しない大量売買を防止。フラッシュクラッシュ等への防御。
**代替案**: 手動確認のみ（自動化時に危険）、閾値なし（暴走リスク）

### 7. テンプレート YAML は init 時に毎回上書き

**選択**: `specs/template.yaml` は `kabu init` で常に上書き。ユーザーの独自戦略は別ファイルで管理。
**理由**: テンプレートは参考用であり、最新版を常に提供したい。ユーザーのカスタム戦略は `config.toml` の `[spec] path` で別ファイルを指定する運用。
**代替案**: 上書きしない（テンプレート更新不可）、`--force` でのみ上書き（ユーザー戦略消失リスク）

## Risks / Trade-offs

- **[CLI 依存]** `cli-claude` / `cli-gemini` は外部 CLI のインストールとログイン状態に依存 → `api-anthropic` / `api-gemini` を代替として提供
- **[J-Quants API 制限]** 無料プランのレート制限・データ範囲制限あり → 1 秒間隔の rate limiting を実装済み
- **[LLM 出力の不安定性]** JSON パース失敗のリスク → markdown コードブロックからの JSON 抽出ロジックを実装済み
- **[Tachibana API 未実装]** execute コマンドは売買シグナル出力のみ、実際の発注はスタブ → API アクセス取得後に実装予定
- **[単一 DB ファイル]** 同時アクセスでのロック競合 → 個人利用のため許容。将来的にはWAL モード検討
