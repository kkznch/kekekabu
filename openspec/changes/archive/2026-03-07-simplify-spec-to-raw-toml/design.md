## Context

`spec.rs` は投資 Spec TOML を `InvestmentSpec` 構造体にデシリアライズしていたが、ユーザーの実際の TOML 構造と構造体の定義が乖離。`serde(default)` により不一致が無視され、ユーザーの戦略パラメータが LLM プロンプトに反映されないバグが発生していた。

Spec の用途は LLM プロンプトへの埋め込みのみで、Rust 側での数値利用はない。

## Goals / Non-Goals

**Goals:**
- ユーザーが自由な TOML 構造で投資 Spec を記述できるようにする
- Spec の内容が確実に LLM プロンプトに反映されるようにする
- 最低限のバリデーション（TOML 構文 + `name` キー存在）を維持する

**Non-Goals:**
- Rust コードでの数値フィルタリング機能の追加（将来必要になったら `toml::Value` で部分パースする）
- Spec フォーマットの標準化や必須セクションの定義

## Decisions

### Decision 1: 生テキスト方式を採用
- **選択**: TOML ファイルの生テキストをそのままプロンプトに埋め込む
- **代替案**: 構造体を実際の TOML に合わせて拡張する → ユーザーが構造を変えるたびに Rust コード変更が必要になるため却下
- **理由**: LLM は TOML を問題なく読める。構造に依存しない方が柔軟性が高い

### Decision 2: `name` のみ構造的に抽出
- **選択**: `toml::Table` で TOML をパースし `name` キーだけ取り出す
- **理由**: ログ表示（`Spec (名前): OK`）に必要。それ以外のフィールドは LLM が解釈する

### Decision 3: バリデーションの簡素化
- **選択**: TOML 構文チェック + `name` 存在確認のみ
- **代替案**: `toml::Value` で特定キーの型チェック → 現時点では過剰。将来必要になったら追加
- **理由**: 型付きバリデーション自体がバグの原因だった（デフォルト値で通過）

## Risks / Trade-offs

- **[Risk] 不正な Spec でもバリデーションが通る** → LLM が解釈できれば実害なし。将来的に `toml::Value` での部分チェックで拡張可能
- **[Trade-off] Rust 側での数値利用ができない** → 現時点で需要なし。必要になったら `toml::Value` から部分パースする設計で対応可能
