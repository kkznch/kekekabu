# 引き継ぎ資料: kekestock → 新リポジトリ

## これは何か

個人向け日本株投資支援 CLI ツール `kekestock` (Rust) の再設計にあたり、別リポジトリで新規に実装するための引き継ぎ資料。現実装の知見と、Gemini による再設計案を統合してある。

---

## 背景と経緯

- `kekestock` は Rust 2024 edition の CLI アプリ（バイナリ名: `kkst`）
- 18モジュール・18サブコマンドまで膨らんだが、**機能を盛りすぎた**ためリライトを決定
- 実際に使われているのは一部のコマンドのみ（collect, watchlist, ta あたり）
- `backtester`, `journal`, `alert`, `verifier`, `agent` 等はほぼ未使用
- 旧リポジトリ: https://github.com/kkznch/kekestock

---

## 決定事項

| 項目 | 決定 |
|---|---|
| 言語 | **Rust** (2024 edition)。Result 型チェーンでワークフローを組む |
| DB | **SQLite** (`rusqlite` bundled)。個人ローカル CLI に最適 |
| 市場 | **JP 専用**。US サポート不要 |
| Gemini 連携 | **gemini-cli** (`gemini -p`) でできる範囲。Grounding は CLI 経由で利用 |
| 出力形式 | **デフォルト JSON**。`--format human` で人間向け表示 |
| 自動化 | **全自動パイプライン**。eval → execute は人間の確認なしで実行 |
| 定期実行 | **launchd (macOS) or cron** で定期実行。フォアグラウンドスケジューラは作らない |
| 実行スケジュール | 朝 scan → fetch → eval、ザラ場開始後 execute、夕方 report の3ジョブ（※バイナリ名は新リポジトリで決定） |
| ストップロス | **証券会社の逆指値注文に委ねる**。execute 時にストップロス付き注文を出す。CLI 側でザラ場中の価格監視はしない（立花証券 API の逆指値対応要確認） |
| パイプライン失敗時 | 朝の `scan → fetch → eval` は `&&` チェーンで途中失敗なら後続を実行しない。`execute` は当日の eval 結果のみ対象とし、当日分がなければ何もせず終了（古い eval で発注しない） |
| 投資 Spec | **YAML で外部定義**。config で Spec ファイルパスを切り替え可能にし、戦略の差し替えを柔軟に |
| データ保持 | **全中間データを DB に保存**。収集データ・分析結果・判断根拠すべて残す |
| Spec バージョン管理 | eval 時に Spec ファイルの **SHA256 ハッシュ** を evaluations テーブルに記録。戦略切替後も過去の判断がどのルールに基づくか追跡可能に |
| fetch スナップショット | fetch_results に URL だけでなく **本文テキストのスナップショット** も保存。ニュースは消えるため、判断根拠の再現性に必須 |
| 買付余力チェック | execute 前に立花 API から **残高・買付余力を取得**。Spec の `max_position_size` を具体的な株数に変換するために必要 |
| サーキットブレーカー | 全銘柄暴落（日経平均の急落等）や異常価格（前日比30%以上乖離）を検知したら **execute を自動停止**。全自動パイプラインの安全装置 |
| ログ | **`tracing` クレートで構造化ログ**をファイル出力。launchd/cron 実行時のデバッグに必須。stdout の JSON 出力とは別 |

## 新設計: 5コマンドパイプライン

Gemini と相談して決めた新アーキテクチャ。「検索の Gemini」と「推論の Claude」を使い分ける。

```
scan → fetch → eval → execute → report
```

| コマンド | 責務 | LLM |
|---|---|---|
| `scan` | J-Quants からデータ取得 + TA 計算でスクリーニング | 不要 |
| `fetch` | 対象銘柄の最新情報を収集・構造化 | **Gemini** (gemini-cli) |
| `eval` | 投資判断（Buy/Hold/Avoid）とスコアリング | **Claude** (claude-cli or API) |
| `execute` | 証券会社 API 経由で注文発行 | 不要 |
| `report` | 判断根拠を Markdown で出力 | 不要 |

### scan の動作

- J-Quants API で株価データを取得して prices テーブルに保存
- TA エンジンでテクニカル指標を計算（RSI, MACD, BB 等）
- スクリーニング条件に合致する銘柄を抽出
- **データ取得と TA 計算を1コマンドで実行**（旧実装の `collect` + `screen` を統合）
- 初回は30日分取得、DB に既存データがあっても差分取得の最適化は後回し（INSERT OR IGNORE で冪等）

### LLM 役割分担の意図

- **Gemini**: TDnet 適時開示 PDF、ニュース、SNS センチメント、競合動向を広く集めて JSON に構造化。長大コンテキスト + Google 検索統合が強み
- **Claude**: Gemini が集めたデータ + TA 指標 + 投資 Spec を食わせて「機関投資家の投資委員会」として判断。論理的整合性と指示忠実度が強み

### 投資 Spec（詳細は `.keke/gemini2.md` 参照）

投資判断ロジックを YAML の「Spec」として外部定義し、Claude に厳守させる。

- config.toml で Spec ファイルパスを指定 → 戦略の差し替えが設定変更だけで可能
- Spec の中身: ユニバースフィルタ（流動性、財務健全性）、定量スコアリング（PBR, PER, 配当利回り, ROE）、定性分析ガイドライン、執行ロジック（ポジションサイジング、ストップロス、トレーリングストップ）
- Claude への出力形式: `{ decision: "Buy|Hold|Avoid", score: 0-100, rationale: {...}, execution_advice: {...} }`
- 具体的な Spec 例は `.keke/gemini2.md` の YAML を参照

## 旧実装から引き継ぐべきもの

### 1. テクニカル分析エンジン（最重要）

旧: `src/screener/indicators.rs` → 新: `scan` コマンドのバックエンド

- RSI (14), MACD, ボリンジャーバンド (20), SMA (5/25/75), EMA (12/26), ATR (14), 出来高MA (20)
- `rust_ti` crate でバルク計算
- データ不足時は空 Vec を返す（パニックしない安全設計）
- `TechnicalIndicators::latest_values()` で HashMap に最新値を集約
- クロス検出: ゴールデンクロス、デッドクロス、MACD シグナルクロス、BB ブレイクアウト
- **そのまま移植可能**。外部依存は `rust_ti` のみ

### 2. LLM バックエンド抽象化

旧: `src/llm/` → 新: そのまま使える

```rust
#[async_trait]
pub trait LlmBackend: Send + Sync {
    async fn send_message(&self, prompt: &str, max_tokens: u32) -> Result<String>;
}
```

- Factory: `create_backend("api-anthropic" | "cli-claude" | "cli-gemini", api_key, model)`
- 3実装: Anthropic HTTP API, Claude CLI (`claude -p`), Gemini CLI (`gemini -p`)
- 用途別に切り替え可能（config の `[llm]` セクション）
- **改善点**: `async-trait` → Rust 2024 の native async trait に移行可能

### 3. LLM プロンプト設計

旧: `src/predictor/prompt.rs`, `src/sentiment/prompt.rs`

- JSON 応答を強制するプロンプト構成
- Markdown コードブロック内の JSON 抽出（`find('{')` ~ `rfind('}')`）
- 構造化パース: `PredictionResponse`, `SentimentResponse`
- **新設計では投資 Spec (YAML) をプロンプトに埋め込む形に進化させる**

### 4. DB スキーマ設計パターン

旧: `src/db/schema.rs` (SQLite, 12テーブル)

再利用すべき設計判断:
- **金額は TEXT 型** → `rust_decimal::Decimal` との相互変換で浮動小数点の精度劣化なし
- **`UNIQUE` 制約 + `INSERT OR IGNORE`** → 冪等なデータ収集（同じデータを何度入れても壊れない）
- **`stocks` テーブルを正規化の中心** → 全テーブルが `stock_id` で FK 参照
- **日時は TEXT 型 (ISO 8601)**

新設計で必要なコアテーブル（12 → 7に絞る）:
1. `stocks` — 銘柄マスタ (ticker, name, market, sector)
2. `prices` — 日足株価 (OHLCV, UNIQUE(stock_id, date))
3. `watchlist` — 監視銘柄
4. `fetch_results` — fetch で収集した中間データ（ニュース、適時開示、センチメント等）。eval の入力になり、判断根拠の追跡にも使う。UNIQUE(url) + INSERT OR IGNORE で冪等収集
5. `evaluations` — eval 結果（旧 predictions + prediction_factors を統合）。score, decision, rationale を含む
6. `portfolio_positions` — ポジション（加重平均コスト）
7. `trades` — 売買履歴（P&L 付き）

### 5. ポートフォリオ管理ロジック

旧: `src/portfolio/mod.rs`

- 買い増し時の加重平均コスト自動計算: `new_avg = (old_avg * old_qty + buy_price * buy_qty) / new_qty`
- 売却時の P&L 自動計算: `pnl = (sell_price - avg_cost) * sell_qty`
- **トランザクション**で position + trade を原子的に更新
- 全売却で `is_active = 0` に自動切替
- `rust_decimal::Decimal` で金額精度を保証

### 6. リスク管理ロジック

旧: `src/risk/mod.rs` + 新: 投資 Spec の execution_logic

- ポジションサイジング: Spec の `max_position_size: 0.05`（資金の5%上限）
- ストップロス: Spec の `stop_loss: -0.07`（-7%で機械的損切り）
- トレーリングストップ: Spec の `trailing_stop: 0.15`（高値から15%下落で利確）
- 旧実装の ATR ベース計算、リスクリワード比計算も参考になる

### 7. 外部 API クライアント

| API | 旧ファイル | 用途 | 新設計での位置 |
|---|---|---|---|
| J-Quants V2 | `src/collector/jquants.rs` | 日足株価 + 上場銘柄マスタ | `scan` のデータソース |
| Google News RSS | `src/collector/news.rs` | ニュース取得（API キー不要） | `fetch` の補助ソース |
| 立花証券 e支店 API | 未実装 | 注文発行 | `execute` で新規実装 |

- 立花証券 API ドキュメント: https://www.e-shiten.jp/e_api/mfds_json_api_menu.html
- **JP 市場専用**。US 市場サポートは不要

### 8. 設定管理パターン

旧: `src/config.rs`

- TOML 設定ファイル (`~/.config/kkst/config.toml`)
- 環境変数で上書き可能 (`JQUANTS_API_KEY`, `ANTHROPIC_API_KEY` 等)
- `AppConfig::require_key()` で必須キーの有無チェックとエラーメッセージ
- serde Deserialize + Default でセクション丸ごとオプショナル

新設計での config 構成案:
```toml
[api]
jquants_api_key = "..."
anthropic_api_key = "..."
tachibana_user = "..."
tachibana_password = "..."

[llm]
fetch = "cli-gemini"       # scan 後の情報収集
eval = "cli-claude"        # 投資判断
eval_model = "..."         # オプション: モデル指定

[spec]
path = "specs/jp-core-value-quality-v1.yaml"  # 投資 Spec ファイルパス

[output]
default_format = "json"    # "json" | "human"
```

### 9. テスト戦略

- 全テスト: インメモリ SQLite (`Connection::open_in_memory()`)
- API モック不要: LLM テストはプロンプト構築 + JSON パースの単体テストのみ
- 外部依存ゼロで CI が高速（120テスト、< 1秒）
- **この戦略はそのまま踏襲すべき**

## 引き継がないもの（旧実装の反省）

### 構造面
- **18コマンドは過剰** → 5コマンドに絞る
- **`main.rs` のファットコントローラー** — 巨大 match 文で全コマンドをハンドル → コマンドハンドラを各モジュールに委譲すべき
- **`display.rs` が全モジュールに散在** — デフォルト JSON、`--format human` でテーブル表示に統一
- **Agent (フォアグラウンドスケジューラ) と Report (cron) の二重構成** → cron で十分
- **Manager 間の暗黙的依存** — SignalManager が内部で WatchlistManager を生成する等
- **Human-in-the-loop 前提の設計** → 全自動パイプラインに変更

### 未使用/過剰機能
- `backtester` — 使われていない
- `journal` — 使われていない
- `alert` — 使われていない
- `verifier` — signal verify と機能重複
- `agent` — cron で代替可能

## 技術スタック（推奨）

### 継続
| クレート | 用途 |
|---|---|
| `clap` (derive) | CLI パーサ |
| `tokio` + `tokio-rusqlite` | async ランタイム + SQLite |
| `rust_decimal` | 金額計算 |
| `rust_ti` | TA 指標計算 |
| `reqwest` | HTTP クライアント |
| `serde` + `serde_json` + `toml` | 直列化 |
| `chrono` | 日時処理 |
| `anyhow` + `thiserror` | エラーハンドリング |
| `which` | CLI バックエンド検出 |

### 再考 / 変更候補
| 旧 | 候補 | 理由 |
|---|---|---|
| `colored` + `tabled` | そのまま or `ratatui` | TUI にするなら ratatui、CLI のままなら継続 |
| `async-trait` | native async trait | Rust 2024 edition なら不要の可能性 |
| `tokio-rusqlite` | `sqlx` (sqlite) | マイグレーション機能が欲しいなら sqlx |

### 新規
- `serde_yaml` — 投資 Spec の YAML パース
- `sha2` — Spec ファイルのハッシュ化（evaluations に記録）
- `tracing` + `tracing-subscriber` — 構造化ログ（ファイル出力、launchd/cron デバッグ用）
- シャープレシオ等の定量評価指標 → `report` コマンドで実装

## データフロー（新設計）

```
[scan]
  J-Quants V2 → stocks + prices (DB)
  TA エンジン → スクリーニング結果（RSI, MACD, BB 等）
       ↓
[fetch]
  gemini-cli → ニュース, 適時開示, SNS センチメント
  Google News RSS → 補助ニュースソース
  → 構造化 JSON で fetch_results (DB) に保存
       ↓
[eval]
  claude-cli ← TA 指標 + fetch_results (DB) + 投資 Spec (YAML)
  → Buy/Hold/Avoid 判断 + スコア + 根拠 → evaluations (DB)
       ↓
[execute] ← 全自動（eval の decision が Buy のものを自動発注）
  立花証券 e支店 API → 注文発行
  → portfolio_positions, trades (DB)
       ↓
[report]
  evaluations + trades → Markdown 出力（Obsidian 等向け）
  シャープレシオ等の定量評価
```

## 最初の MVP スコープ

全部一気に作らない。以下の順で段階的に:

### Phase 1: scan + eval（最小動作）
- J-Quants からデータ取得 + TA 計算 → Claude で eval
- watchlist 管理
- DB は stocks, prices, watchlist, evaluations の4テーブル
- Gemini 連携なし（fetch は後回し、ニュースは Google News RSS で代用）
- 出力: JSON デフォルト + `--format human`

### Phase 2: fetch + report
- gemini-cli でリッチな情報収集 → fetch_results テーブル
- 投資 Spec の YAML 外部化 + config で切り替え
- Markdown レポート出力

### Phase 3: execute
- 立花証券 API 連携
- ポートフォリオ管理 + P&L 追跡
- サーキットブレーカー（Spec の execution_logic に基づく）
- 全自動パイプライン完成: scan → fetch → eval → execute
